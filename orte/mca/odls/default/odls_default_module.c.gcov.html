      30             :  * There is a complicated sequence of events that occurs when the
      31             :  * parent forks a child process that is intended to launch the target
      32             :  * executable.
      33             :  *
      34             :  * Before the child process exec's the target executable, it might tri
      35             :  * to set the affinity of that new child process according to a
      36             :  * complex series of rules.  This binding may fail in a myriad of
      37             :  * different ways.  A lot of this code deals with reporting that error
      38             :  * occurately to the end user.  This is a complex task in itself
      39             :  * because the child process is not "really" an ORTE process -- all
      40             :  * error reporting must be proxied up to the parent who can use normal
      41             :  * ORTE error reporting mechanisms.
      42             :  *
      43             :  * Here's a high-level description of what is occurring in this file:
      44             :  *
      45             :  * - parent opens a pipe
      46             :  * - parent forks a child
      47             :  * - parent blocks reading on the pipe: the pipe will either close
      48             :  *   (indicating that the child successfully exec'ed) or the child will
      49             :  *   write some proxied error data up the pipe
      50             :  *
      51             :  * - the child tries to set affinity and do other housekeeping in
      52             :  *   preparation of exec'ing the target executable
      53             :  * - if the child fails anywhere along the way, it sends a message up
      54             :  *   the pipe to the parent indicating what happened -- including a
      55             :  *   rendered error message detailing the problem (i.e., human-readable).
      56             :  * - it is important that the child renders the error message: there
      57             :  *   are so many errors that are possible that the child is really the
      58             :  *   only entity that has enough information to make an accuate error string
      59             :  *   to report back to the user.
      60             :  * - the parent reads this message + rendered string in and uses ORTE
      61             :  *   reporting mechanisms to display it to the user
      62             :  * - if the problem was only a warning, the child continues processing
      63             :  *   (potentially eventually exec'ing the target executable).
      64             :  * - if the problem was an error, the child exits and the parent
      65             :  *   handles the death of the child as appropriate (i.e., this ODLS
      66             :  *   simply reports the error -- other things decide what to do).
      67             :  */
