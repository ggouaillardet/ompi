    1752             : /*
    1753             :    malloc_chunk details:
    1754             : 
    1755             :     (The following includes lightly edited explanations by Colin Plumb.)
    1756             : 
    1757             :     Chunks of memory are maintained using a `boundary tag' method as
    1758             :     described in e.g., Knuth or Standish.  (See the paper by Paul
    1759             :     Wilson ftp://ftp.cs.utexas.edu/pub/garbage/allocsrv.ps for a
    1760             :     survey of such techniques.)  Sizes of free chunks are stored both
    1761             :     in the front of each chunk and at the end.  This makes
    1762             :     consolidating fragmented chunks into bigger chunks very fast.  The
    1763             :     size fields also hold bits representing whether chunks are free or
    1764             :     in use.
    1765             : 
    1766             :     An allocated chunk looks like this:
    1767             : 
    1768             : 
    1769             :     chunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1770             :             |             Size of previous chunk, if allocated            | |
    1771             :             +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1772             :             |             Size of chunk, in bytes                         |P|
    1773             :       mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1774             :             |             User data starts here...                          .
    1775             :             .                                                               .
    1776             :             .             (malloc_usable_space() bytes)                     .
    1777             :             .                                                               |
    1778             : nextchunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1779             :             |             Size of chunk                                     |
    1780             :             +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1781             : 
    1782             : 
    1783             :     Where "chunk" is the front of the chunk for the purpose of most of
    1784             :     the malloc code, but "mem" is the pointer that is returned to the
    1785             :     user.  "Nextchunk" is the beginning of the next contiguous chunk.
    1786             : 
    1787             :     Chunks always begin on even word boundries, so the mem portion
    1788             :     (which is returned to the user) is also on an even word boundary, and
    1789             :     thus at least double-word aligned.
    1790             : 
    1791             :     Free chunks are stored in circular doubly-linked lists, and look like this:
    1792             : 
    1793             :     chunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1794             :             |             Size of previous chunk                            |
    1795             :             +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1796             :     `head:' |             Size of chunk, in bytes                         |P|
    1797             :       mem-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1798             :             |             Forward pointer to next chunk in list             |
    1799             :             +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1800             :             |             Back pointer to previous chunk in list            |
    1801             :             +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1802             :             |             Unused space (may be 0 bytes long)                .
    1803             :             .                                                               .
    1804             :             .                                                               |
    1805             : nextchunk-> +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1806             :     `foot:' |             Size of chunk, in bytes                           |
    1807             :             +-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
    1808             : 
    1809             :     The P (PREV_INUSE) bit, stored in the unused low-order bit of the
    1810             :     chunk size (which is always a multiple of two words), is an in-use
    1811             :     bit for the *previous* chunk.  If that bit is *clear*, then the
    1812             :     word before the current chunk size contains the previous chunk
    1813             :     size, and can be used to find the front of the previous chunk.
    1814             :     The very first chunk allocated always has this bit set,
    1815             :     preventing access to non-existent (or non-owned) memory. If
    1816             :     prev_inuse is set for any given chunk, then you CANNOT determine
    1817             :     the size of the previous chunk, and might even get a memory
    1818             :     addressing fault when trying to do so.
    1819             : 
    1820             :     Note that the `foot' of the current chunk is actually represented
    1821             :     as the prev_size of the NEXT chunk. This makes it easier to
    1822             :     deal with alignments etc but can be very confusing when trying
    1823             :     to extend or adapt this code.
    1824             : 
    1825             :     The two exceptions to all this are
    1826             : 
    1827             :      1. The special chunk `top' doesn't bother using the
    1828             :         trailing size field since there is no next contiguous chunk
    1829             :         that would have to index off it. After initialization, `top'
    1830             :         is forced to always exist.  If it would become less than
    1831             :         MINSIZE bytes long, it is replenished.
    1832             : 
    1833             :      2. Chunks allocated via mmap, which have the second-lowest-order
    1834             :         bit (IS_MMAPPED) set in their size fields.  Because they are
    1835             :         allocated one-by-one, each must contain its own trailing size field.
    1836             : 
    1837             : */

    1986             : /*
    1987             :   Bins
    1988             : 
    1989             :     An array of bin headers for free chunks. Each bin is doubly
    1990             :     linked.  The bins are approximately proportionally (log) spaced.
    1991             :     There are a lot of these bins (128). This may look excessive, but
    1992             :     works very well in practice.  Most bins hold sizes that are
    1993             :     unusual as malloc request sizes, but are more usual for fragments
    1994             :     and consolidated sets of chunks, which is what these bins hold, so
    1995             :     they can be found quickly.  All procedures maintain the invariant
    1996             :     that no consolidated chunk physically borders another one, so each
    1997             :     chunk in a list is known to be preceeded and followed by either
    1998             :     inuse chunks or the ends of memory.
    1999             : 
    2000             :     Chunks in bins are kept in size order, with ties going to the
    2001             :     approximately least recently used chunk. Ordering isn't needed
    2002             :     for the small bins, which all contain the same-sized chunks, but
    2003             :     facilitates best-fit allocation for larger chunks. These lists
    2004             :     are just sequential. Keeping them in order almost never requires
    2005             :     enough traversal to warrant using fancier ordered data
    2006             :     structures.

    2009             :     recently freed at the front, and allocations are taken from the
    2010             :     back.  This results in LRU (FIFO) allocation order, which tends
    2011             :     to give each chunk an equal opportunity to be consolidated with
    2012             :     adjacent freed chunks, resulting in larger free chunks and less
    2013             :     fragmentation.

    2016             :     as a malloc_chunk. This avoids special-casing for headers.
    2017             :     But to conserve space and improve locality, we allocate
